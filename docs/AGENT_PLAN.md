## 1. 에이전트 개요

7개의 전문 에이전트가 순차 및 병렬로 협업하여 고등학생의 수행평가 보고서 작성을 지원하는 멀티에이전트 시스템이다. 모든 보고서는 AI/인공지능 진로와 자연스럽게 연결되며, 학생이 직접 참여하는 구간을 명확히 구분한다.

**대상**: 고등학교 2학년, AI/인공지능 학과 진학 희망 학생
**핵심 원칙**: AI가 작성한 티가 나지 않을 것 / 진로 연결은 자연스럽게 / 교육과정 범위 준수 / 참고 문헌은 실제 자료만

### 1.1 에이전트 목록

| # | 에이전트명 | 유형 | 정의파일 | 역할 | 입력 | 출력 |
|---|-----------|------|---------|------|------|------|
| 00 | Orchestrator | Core | `agents/orchestrator.md` | 전체 워크플로우 관리, 단계 전환, 품질 관리 | 학생 요청, `profile.yaml`, 보고서 기록 | 단계별 결과 요약, 단계 전환 지시 |
| 01 | Topic Scout | Core | `agents/topic_scout.md` | AI 진로 연결 가능한 주제 5개+ 제안 | 과목, 범위, 실험 여부, 키워드 이력 | 주제 목록 (개념, AI 연결, 난이도 등) |
| 02 | Research Agent | Core | `agents/research_agent.md` | 참고 문헌/논문/도서 조사 및 이론 배경 정리 | 선택 주제, 과목/단원, AI 키워드 | 이론 정리, 참고 도서/논문/기사 |
| 03 | Experiment Designer | Core | `agents/experiment_designer.md` | 가설, 변인, 실험 절차, 데이터 기록 양식 설계 | 선택 주제, 이론 배경 (Agent 02) | 가설, 변인표, 절차, 기록 양식 |

## 2. 에이전트 상세 명세

### 2.1 Orchestrator (Agent 00) - 오케스트레이터

| 항목 | 내용 |
|------|------|
| **역할** | 전체 워크플로우 관리, 단계 전환, 품질 관리 |
| **입력** | 학생 요청 (과목, 범위, 실험 여부), `config/profile.yaml`, 기존 보고서 기록 (`reports/`) |
| **출력** | 각 단계별 결과 요약, 단계 전환 지시, 최종 완료 보고 |

**세부 동작 규칙**:
- 세션 시작 시 `config/profile.yaml`을 읽고, 기존 보고서 기록과 AI 키워드 사용 이력(`reports/keyword_tracker.yaml`)을 확인
- 각 단계 완료 시 결과를 요약하여 학생에게 제시하고, 학생 확인 후 다음 단계로 전환
- 각 단계 결과를 `reports/{과목}_{주제}/` 폴더에 저장
- 교육과정 범위 이탈, AI 연결의 억지스러움, 학생 수행 가능성을 지속 점검

---

### 2.2 Topic Scout (Agent 01) - 주제 발굴 에이전트

| 항목 | 내용 |
|------|------|
| **역할** | 과목 범위 안에서 AI 진로와 연결 가능한 실험/탐구 주제 5개 이상 제안 |
| **입력** | 과목명, 단원/범위, 실험 필수 여부, (선택) 이전 과목 사용 키워드 목록 |
| **출력** | 주제 목록 (주제명, 교과 핵심 개념, AI 연결 고리, 실험 가능 여부, 실험 개요, 난이도, 진로 어필 포인트) |

**규칙**: 교육과정 범위 내에서만 제안 / AI 연결이 자연스러울 것 / 이전 키워드와 겹치되 동일 반복은 회피 / 주제 간 난이도와 방향의 다양성 확보

---

### 2.3 Research Agent (Agent 02) - 자료 조사 에이전트

| 항목 | 내용 |
|------|------|
| **역할** | 선택된 주제에 대한 참고 문헌, 도서, 논문, AI 관련 자료 조사 및 이론 배경 정리 |
| **입력** | 선택된 주제, 과목 및 단원, AI 연결 키워드 |
| **출력** | 교과서 기반 이론 정리, 참고 도서 2-3권, 참고 논문/기사 2-3건, AI 활용 참고 자료 1-2건 |

**규칙**: 실제 존재하는 자료만 추천 (가짜 문헌 절대 금지) / 고등학생 수준 자료 우선 / 한국어 자료 우선 / 필요 시 WebSearch 도구 활용

---

### 2.4 Experiment Designer (Agent 03) - 실험 설계 에이전트

| 항목 | 내용 |
|------|------|
| **역할** | 가설 수립, 변인 설정, 실험 절차, 데이터 기록 양식, AI 관점 연결 설계 |
| **입력** | 선택된 주제, 이론 배경 (Agent 02 결과), 학교 실험실 환경 (기본 장비 가정) |
| **출력** | 탐구 질문, 가설, 변인 설정표(독립/종속/통제), 준비물 목록, 단계별 실험 절차, 데이터 기록 양식, AI 관점 연결 서술 |

**규칙**: 고등학교 실험실에서 실제 수행 가능 / 1-2교시 내 완료 가능 규모 / 안전사고 위험 최소화 / 정량적 데이터 수집 가능한 설계 우선

---

### 2.5 Report Writer (Agent 04) - 보고서 작성 에이전트

| 항목 | 내용 |
|------|------|
| **역할** | 수집된 모든 자료를 종합하여 보고서 뼈대(구조 + 내용) 작성 |
| **입력** | 주제 및 이론 배경, 참고 문헌 목록, 실험 설계 (있는 경우), 실험 결과 데이터 (학생 제공), 학생 프로필 |
| **출력** | 보고서 전체 (제목, 탐구 동기, 이론적 배경, 탐구 방법, 탐구 결과, 결론 및 고찰, 참고 문헌) |

**규칙**: 학생 참여 부분에 `[본인 경험 추가]`, `[본인 말로 다듬기]` 등 마커 표시 / 고찰에 궁금한 점/아쉬운 점 포함 / 실험 실패나 예상과 다른 결과 반영 / AI 진로 연결은 결론에서 1-2문장으로 자연스럽게

---

### 2.6 Voice Coach (Agent 05) - 어투 코치 에이전트

| 항목 | 내용 |
|------|------|
| **역할** | 보고서의 AI스러운 표현을 점검하고, 학생 어투로 변환하는 수정 가이드 제공 (직접 수정 금지) |
| **입력** | 보고서 초안 전체 텍스트 |
| **출력** | AI 느낌 체크리스트 (8개 항목), 항목별 수정 제안 (원문/문제/수정 방향), 학생다움 추가 포인트 |

**점검 항목**: 과도하게 정돈된 문장 / 균일한 문단 길이 / "~라고 할 수 있다" 반복 / 접속사 남발 / 감정 부재 / 너무 완벽한 논리 / 이모지/과도한 강조 / 교과서 문장 복붙 느낌

**규칙**: 절대 직접 수정본을 제공하지 않음 / 방향만 제시하여 학생이 직접 작성하도록 유도 / 약간의 불완전함이 자연스러움임을 안내

---

### 2.7 Topic Advisor (Agent 06) - 주제 심화 토론 에이전트

| 항목 | 내용 |
|------|------|
| **역할** | 주제 후보들에 대한 다면 평가 (난이도/임팩트/AI 연결/실현가능성/면접 설명력/성장 서사) 및 조합/변형 제안 |
| **입력** | 주제 후보 목록 (Agent 01 결과), 학생의 반응/선호, 프로그램/과제 맥락 (수행평가/프로그램 지원/대회) |
| **출력** | 주제별 다면 평가 (6개 기준, 각 1-5점), 주제 조합/변형 제안, 맥락별 최종 추천 및 근거 |

**규칙**: 학생 반응 존중하되 근거 있는 반론 제시 / "안 돼"보다 "이렇게 바꾸면 더 나아" 방식 / 최종 선택은 학생이 결정 / 맥락에 따라 평가 기준 가중치 차등 적용

---

### 2.8 Interview Prep (Agent 07) - 면접 준비 에이전트

| 항목 | 내용 |
|------|------|
| **역할** | 보고서 기반 예상 질문, 진로 연결 질문, 돌발/심화 질문 준비 및 면접 시뮬레이션 제공 |
| **입력** | 제출 보고서 전문, 학생 프로필 (진로, 배경 지식 수준), 프로그램 성격 |
| **출력** | 보고서 기반 예상 질문 5-7개, 진로 연결 질문 3-5개, 돌발/심화 질문 3-5개, 5분 면접 시뮬레이션 시나리오 |

**규칙**: 외울 문장이 아닌 포인트만 제공 / "모른다"는 답변도 전략임을 안내 / AI 사용 의심을 살 수 있는 답변 패턴 경고 / 면접관 관점에서 "뽑고 싶은 이유"가 답변에 담기도록 설계

```
         │  Voice Coach     │  입력: 보고서 초안
         │  (Agent 05)      │  출력: AI스러움 체크리스트, 수정 가이드
         └────────┬────────┘
                  ▼
         ┌─────────────────────────┐
         │ [학생: 본인 어투로 다듬기] │
         └────────┬────────────────┘
                  ▼
         ┌─────────────────┐
         │  Interview Prep  │  입력: 최종 보고서, 프로필, 프로그램 성격

```

## 4. 실행 순서

### 4.1 실험 포함 보고서 (전체 파이프라인)

```
Phase 1 [순차]  Orchestrator → Topic Scout → Topic Advisor → [학생 주제 선택]
Phase 2 [순차]  Research Agent → Experiment Designer → [학생 실험 수행]
Phase 3 [순차]  Report Writer → Voice Coach → [학생 최종 다듬기]
Phase 4 [순차]  Interview Prep
```

| 순서 | Phase | 에이전트 | 실행 방식 | 학생 개입 |
|------|-------|----------|-----------|-----------|
| 1 | Phase 1 | Orchestrator (00) | 순차 | 과목/범위/실험 여부 입력 |
| 2 | Phase 1 | Topic Scout (01) | 순차 | - |
| 3 | Phase 1 | Topic Advisor (06) | 순차 | 선호/반응 전달 |
| 4 | Phase 1 | - | **학생 대기** | 주제 최종 선택 |
| 5 | Phase 2 | Research Agent (02) | 순차 | - |
| 6 | Phase 2 | Experiment Designer (03) | 순차 | - |
| 7 | Phase 2 | - | **학생 대기** | 실험 수행 후 결과 데이터 입력 |
| 8 | Phase 3 | Report Writer (04) | 순차 | - |
| 9 | Phase 3 | Voice Coach (05) | 순차 | - |
| 10 | Phase 3 | - | **학생 대기** | 본인 어투로 최종 다듬기 |
| 11 | Phase 4 | Interview Prep (07) | 순차 | - |

### 4.2 실험 없는 보고서 (축소 파이프라인)

```
Phase 1 [순차]  Orchestrator → Topic Scout → Topic Advisor → [학생 주제 선택]
Phase 2 [순차]  Research Agent → Report Writer → Voice Coach → [학생 최종 다듬기]
Phase 3 [순차]  Interview Prep
```

순서 6(Experiment Designer)과 순서 7(학생 실험 수행)이 생략되며, Report Writer는 Research Agent 결과만으로 보고서를 작성한다.

### 4.3 병렬 실행 가능 구간

현재 설계에서 각 에이전트는 이전 에이전트의 출력에 의존하므로 **기본적으로 순차 실행**이다. 다만, 아래 구간은 향후 병렬 실행이 가능하다:

| 병렬 가능 조합 | 실행 방식 | 조건 | 비고 |
|---------------|-----------|------|------|
| Topic Scout (01) + Topic Advisor (06) | 병렬 | 01이 주제를 하나씩 스트리밍하면 06이 즉시 평가 시작 | 현재 미구현 |
| Research Agent (02) + Experiment Designer (03) | 병렬 | 03이 이론 배경 없이 기본 설계 가능한 경우 | 품질 저하 우려로 비권장 |
| Voice Coach (05) + Interview Prep (07) | 병렬 | 07이 초안 기반으로 시작하고, 05의 수정 가이드를 나중에 반영 | 현재 미구현 |

## 5. 에러 핸들링

### 5.1 에이전트별 실패 시 대응

| 에이전트 | 실패 유형 | 대응 전략 |
|----------|-----------|-----------|
| **Orchestrator (00)** | 프로필 파일 누락 | 기본값으로 초기화 (고2, AI 진로). 학생에게 프로필 정보 직접 확인 |
| | keyword_tracker.yaml 누락 | 빈 이력으로 시작. 첫 보고서로 간주 |
| **Topic Scout (01)** | 주제 5개 미달 | 범위를 인접 단원까지 확장하여 재시도. 그래도 부족하면 3개 이상으로 하한 완화 |
| | 교육과정 범위 이탈 | Orchestrator가 범위 체크 후 해당 주제 제거, 재생성 요청 |
| **Research Agent (02)** | WebSearch 실패/타임아웃 | 오프라인 모드로 전환. 교과서 기반 이론 정리만으로 진행하고, 참고 자료는 학생에게 도서관 검색 안내 |
| | 가짜 문헌 생성 의심 | Orchestrator가 출처 검증 요청. 검증 불가 자료는 제거 후 재조사 |
| **Experiment Designer (03)** | 실현 불가능한 설계 | 장비/시약 대체재 제안으로 재설계. 대체 불가 시 실험 범위 축소 |
| | Research Agent 결과 부족 | 최소 이론 배경으로 기본 실험 설계 시도. 품질 미달 시 02 재실행 |
| **Report Writer (04)** | 학생 실험 데이터 미입력 | 데이터 기록 양식(03 출력)을 다시 안내하고 대기. 가상 데이터로 작성하지 않음 |
| | 선행 에이전트 결과 불완전 | 누락 섹션을 `[추가 자료 필요]`로 표시하고, 해당 에이전트 재실행 여부를 Orchestrator에 보고 |
| **Voice Coach (05)** | 보고서 초안 누락 | Report Writer 재실행 요청. 초안 없이는 동작 불가 |
| | 모든 항목 통과 (점검 결과 없음) | 결과를 그대로 전달하되, Orchestrator가 "너무 완벽한 것은 의심 대상"으로 2차 점검 |
| **Topic Advisor (06)** | 학생 반응 미입력 | 반응 없이 객관적 평가만 제공. 최종 추천은 보류하고 학생 의견 대기 |
| | 모든 주제 낮은 점수 | Topic Scout에 범위 확장 재요청. 또는 기존 주제의 변형/조합 제안 |
| **Interview Prep (07)** | 최종 보고서 미완성 | Report Writer 초안 기반으로 임시 질문 생성. 최종본 완성 후 업데이트 |

### 5.2 공통 에러 처리 원칙

1. **롤백 우선**: 에이전트 실패 시 이전 단계로 돌아가 재시도. 다음 단계로 강제 진행하지 않음
2. **학생 알림**: 모든 에러 상황은 Orchestrator가 학생에게 알리고, 대안을 제시
3. **부분 결과 보존**: 실패한 에이전트의 부분 결과가 있으면 보존하고, 재실행 시 활용
4. **최대 재시도**: 동일 에이전트 재실행은 최대 2회. 2회 초과 시 해당 단계를 수동 처리로 전환
5. **결과 저장**: 각 에이전트의 출력은 즉시 `reports/{과목}_{주제}/` 폴더에 저장하여 복구 가능

## 6. 절대 규칙

| ⛔ | 규칙 | 설명 |
|----|------|------|
| ⛔ | AI 작성 티 금지 | AI가 작성한 것처럼 보이는 표현, 패턴, 구조를 사용하지 않는다 |
| ⛔ | 가짜 문헌 금지 | 실제 존재하지 않는 도서, 논문, 기사를 참고 자료로 제시하지 않는다 |
| ⛔ | 교육과정 이탈 금지 | 해당 과목의 교육과정 범위를 벗어나는 주제나 내용을 포함하지 않는다 |
| ⛔ | 직접 수정 금지 (Voice Coach) | Voice Coach는 수정본을 직접 제공하지 않고, 방향만 제시한다 |
| ⛔ | 가상 데이터 금지 | 학생이 실험 데이터를 제공하지 않은 상태에서 가상 데이터로 보고서를 작성하지 않는다 |
| ⛔ | 직접 통신 금지 | 에이전트 간 통신은 반드시 Orchestrator를 경유한다 (추적성 확보) |
| ⛔ | 강제 진행 금지 | 에이전트 실패 시 다음 단계로 강제 진행하지 않고, 롤백 후 재시도한다 |

### 8.4 확장 시 고려사항

- 에이전트 수가 10개를 초과하면 Orchestrator의 부하가 증가하므로, **Sub-Orchestrator** 도입 검토 (예: 작성 단계 관리자, 평가 단계 관리자)
- 새 에이전트 추가 시 기존 의존성 그래프에 미치는 영향을 먼저 분석
- 모든 에이전트는 동일한 입출력 형식 규약을 따르도록 표준화 유지
- 에이전트 간 통신은 반드시 Orchestrator를 경유 (직접 통신 금지로 추적성 확보)
